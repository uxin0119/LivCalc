# 프로젝트 코딩 가이드라인

## 파일 크기 제한 및 모듈화 규칙

### 핵심 규칙
**한 파일의 크기가 1000라인을 넘어가면 반드시 모듈화를 진행한다.**

### 모듈화 방법

#### 1. 컴포넌트 분리
- 큰 컴포넌트는 작은 서브 컴포넌트로 분리
- 재사용 가능한 UI 요소는 별도 파일로 추출
- 예시:
  ```
  components/
  ├── Terminal.tsx (메인 컴포넌트)
  ├── TerminalLog.tsx (로그 표시)
  ├── TerminalInput.tsx (입력 처리)
  └── TerminalStatusBar.tsx (상태 표시)
  ```

#### 2. 로직 분리
- 비즈니스 로직은 별도 모듈로 분리
- 관련된 기능끼리 묶어서 관리
- 예시:
  ```
  game-logic/
  ├── combat.ts (전투 로직)
  ├── cards.ts (카드 관련)
  ├── enemies.ts (적 관련)
  └── progression.ts (진행도 관련)
  ```

#### 3. 타입 정의 분리
- 복잡한 타입은 types 폴더로 분리
- 도메인별로 타입 파일 구성
- 예시:
  ```
  types/
  ├── game.ts (게임 핵심 타입)
  ├── ui.ts (UI 관련 타입)
  └── api.ts (API 관련 타입)
  ```

#### 4. 유틸리티 함수 분리
- 재사용 가능한 유틸리티는 별도 파일로
- 기능별로 파일 구성
- 예시:
  ```
  utils/
  ├── format.ts (포맷팅 함수)
  ├── validation.ts (검증 함수)
  └── helpers.ts (헬퍼 함수)
  ```

### 체크리스트

파일을 작성할 때 다음을 확인:
- [ ] 파일이 1000라인을 넘어가는가?
- [ ] 분리 가능한 독립적인 기능이 있는가?
- [ ] 여러 책임을 가지고 있는가?
- [ ] 재사용 가능한 부분이 있는가?

위 항목 중 하나라도 해당되면 모듈화를 고려한다.

### 모듈화 시 주의사항

1. **명확한 책임 분리**: 각 모듈은 하나의 명확한 책임만 가져야 함
2. **의존성 최소화**: 모듈 간 의존성을 최소화
3. **네이밍 일관성**: 파일명과 내부 함수명의 일관성 유지
4. **문서화**: 각 모듈의 역할을 명확히 문서화

### 예외 사항

다음 경우는 1000라인을 초과해도 허용 가능:
- 자동 생성된 파일 (예: 타입 정의 파일)
- 설정 파일
- 테스트 파일 (단, 그룹핑을 고려)

### 리팩토링 우선순위

파일이 1000라인을 넘었을 때:
1. 먼저 중복 코드 제거
2. 유틸리티/헬퍼 함수 분리
3. 컴포넌트 분리
4. 로직 모듈 분리
5. 타입 정의 분리

## 추가 코딩 규칙

### TypeScript 사용
- 모든 코드는 TypeScript로 작성
- any 타입 사용 최소화
- 명확한 타입 정의 필수

### 함수형 프로그래밍 선호
- 순수 함수 작성 지향
- 불변성 유지
- side effect 최소화

### 성능 고려
- 불필요한 리렌더링 방지
- 메모이제이션 활용
- 적절한 데이터 구조 선택

이 가이드라인은 프로젝트의 유지보수성과 확장성을 위한 것입니다.
